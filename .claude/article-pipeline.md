# Article Pipeline

This document is the source of truth for how the article-writing skills cooperate. Each skill `SKILL.md` references this file. If a skill's documentation contradicts what is written here, this file wins — update the skill.

## Skill Chain

```
topic-dialogue → brainstorm-article → write-article → grill-article → enrich-article
                                                                            ↓
                                                                   published article
                                                                            ↓
                                                                   announce-article (terminal-only output)
                                                                            ↓
                                                                   prospect-topics (cross-article, on demand)
```

Each step in the chain consumes a file artifact from the previous step and (where it makes sense) writes a new one. The chain is intentionally file-driven so that any step can be re-run later in a fresh session without losing context, and so that a session can be picked up mid-pipeline.

## Artifact Layout

All intermediate artifacts live under `.article-work/`, a hidden top-level directory. One folder per article session, prefixed with the date so `ls` orders by recency:

```
.article-work/
  {YYYY-MM-DD}-{slug}/
    dialogue.md              ← topic-dialogue
    brief.md                 ← brainstorm-article
    enrichment-notes.md      ← cumulative: brainstorm seeds, write/grill append
    grill-findings.md        ← grill-article
  _prospect/
    {YYYY-MM-DD}.md          ← prospect-topics (cross-article scan)
```

The **published article** itself lives under `mkdocs/docs/blog/posts/` and is the only file that ever ends up next to other published content. Nothing from `.article-work/` is published — it is the workshop floor.

### Naming rules

- `slug` is the URL slug as decided in the brief (e.g. `gateway-pattern`, `opencqrs-no-interceptors`). It is the same slug that ends up in the published article's frontmatter.
- The date prefix is the date the **first** artifact in the session was created. Subsequent skills in the same session reuse the existing folder; they do not rebase the date.
- If a topic is revisited weeks later and a fresh dialogue happens, a **new** folder with the new date is created. Each session is self-contained.

### `.gitignore`

`.article-work/` should be gitignored at the repo root. It is local scratch space. Published articles are the only thing that gets committed.

## Cumulative `enrichment-notes.md`

`enrichment-notes.md` is the most important shared artifact in the chain. It is **cumulative**: more than one skill appends to it, and each appending skill marks its contribution with a section header so the provenance stays visible.

### Contract

| Skill | Action on enrichment-notes.md |
|---|---|
| `topic-dialogue` | Does nothing. (Material lives in `dialogue.md`.) |
| `brainstorm-article` | **Creates** the file with whatever off-scope material surfaced during the brief discussion (sidebar ideas, terminology side notes, design alternatives that did not make it into the brief). |
| `write-article` | **Appends** material that came up _during writing_ — naming debates, comparisons to other frameworks, design decisions the article alludes to but does not unfold. |
| `grill-article` | **Appends** the `Open for Reflection` items from grilling and any `Intentional / Defended` items the author explicitly wants surfaced as sidebars rather than as main-text revisions. |
| `enrich-article` | **Reads** and applies. Does not write back. |

### File structure

```markdown
---
article: <slug>
purpose: Cumulative companion notes for the enrich-article skill. Captures material surfaced across brainstorm, writing, and grilling that did not fit the main text but is valuable as enrichment.
---

# Enrichment Notes: {Article Title}

## From brainstorm-article ({YYYY-MM-DD})

### Collapsible Deep Dives (high priority)
…
### Cross-Link Targets (in addition to those already generated by the standard rules)
…
### Admonitions (recommended)
…
### Abbreviation Tooltips (article-specific, do not duplicate from glossary.md)
…
### Content Annotations ({ .annotate } markers)
…
### External References (for context)
…
### Open Questions / Future-Article Seeds
…
### Code Reference Hints (for content annotations or admonitions)
…
### Style/Voice Notes for the Enricher
…

## From write-article ({YYYY-MM-DD})

### Collapsible Deep Dives (high priority)
…
### Cross-Link Targets …
… (same sections as above; omit empty sections)

## From grill-article ({YYYY-MM-DD})

### Open for Reflection (candidates for sidebars / annotations)
- Finding: …
  Suggested handling: …
### Intentional / Defended (surfaced as sidebars instead of main-text changes)
- …
```

### Rules

- Each contributing skill writes its own dated `## From {skill} ({date})` section. **Never overwrite a previous skill's section.**
- Within a contributor's section, omit subsections that have no content. Do not pad.
- `enrich-article` treats the **union** of contributor sections as the baseline. If two contributors named the same cross-link target, treat it as one — but still mention both in the coverage report.
- If `enrichment-notes.md` does not exist when `enrich-article` runs, fall back to the article alone (legacy mode). Do not fail.

## Handoff Format

Every Skill → Skill invocation passes the path to the **session folder** (or specific artifact when only one applies), plus a brief instruction.

Example, `topic-dialogue` → `brainstorm-article`:

```
Topic: <topic exactly as given by the user>

A topic dialogue transcript has been saved at:
.article-work/{YYYY-MM-DD}-{slug}/dialogue.md

Read that file first. The "Distilled Summary" includes the author's verdict on whether the topic carries.
Continue from the brainstorm-article workflow.
```

The receiving skill is responsible for resolving any other artifacts in the session folder it needs.

## OpenCQRS Terminology (applies to every skill)

- Never use the term **"aggregate"** in OpenCQRS articles, dialogues, briefs, grill findings, or enrichment material. Use **"instance"** or **"state"** instead.
- This applies to skill-generated text, not to user-written input. If the user uses "aggregate" in dialogue, preserve it verbatim in the transcript — but do not adopt it in skill output.

## Skill Responsibilities Summary

| Skill | Reads | Writes | Hands off to |
|---|---|---|---|
| `topic-dialogue` | — (just $ARGUMENTS) | `dialogue.md` | `brainstorm-article` |
| `brainstorm-article` | `dialogue.md` (if present) | `brief.md`, seeds `enrichment-notes.md` | `write-article` |
| `write-article` | `brief.md`, optionally `dialogue.md` | published article in `mkdocs/docs/blog/posts/`, appends to `enrichment-notes.md` | `grill-article` (optional) or `enrich-article` |
| `grill-article` | published article, source code, docs (Modes A+B via a **sub-agent in fresh context**; Mode C dialogue in main session) | `grill-findings.md`, appends to `enrichment-notes.md` | `enrich-article` |
| `enrich-article` | published article, `enrichment-notes.md` | enriched published article (in place) | `announce-article` (optional, after merge) |
| `announce-article` | published article, frontmatter, optionally session folder | — (terminal-only: sheet row fields + 3 LinkedIn-post variants) | — (terminal) |
| `prospect-topics` | all session folders + published articles | `_prospect/{date}.md` | — (terminal, on demand) |
